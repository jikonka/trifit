# TriFit QA 与安全验证要求（强制执行）

> 目的：防止“看起来后端隔离没问题，但前端仍发生数据串号/泄露”的情况再次发生。
> 适用范围：所有涉及认证、个人信息、比赛信息、活动数据、缓存、迁移逻辑的改动。

---

## 1. 事故复盘：为什么之前 QA 没抓到

本次漏检的根因不是 Supabase RLS，而是 **QA 覆盖范围缺失**：

- 现有脚本 `docs/qa-user-isolation.js` 与 `docs/qa-user-isolation-runner.js` 主要是：
  - 直接调用 Supabase REST API；
  - 验证 `user_id` + RLS 的跨用户读写隔离。
- 这两份脚本 **没有覆盖前端状态层**，具体缺失：
  - 没有在浏览器真实执行 `auth.js`；
  - 没有验证 `localStorage` 键是否按用户隔离；
  - 没有测试“同一浏览器 A 登录 -> 登出 -> B 登录”的账号切换路径；
  - 没有覆盖 `migrateLocalStorageData()` 等首登迁移流程；
  - 没有模拟 Supabase 查询失败时的前端 fallback 行为。

**结论**：之前 QA 只证明了“数据库层隔离正确”，没有证明“前端缓存/迁移不会串号”。

---

## 2. QA 入口要求（每次改动必须满足）

每次涉及用户数据的改动，QA 必须同时包含以下三层：

1. **数据库层（RLS）**
   - 继续运行现有隔离脚本，确认跨用户 API 越权不可行。
2. **前端会话层（浏览器真实流程）**
   - 必须做同浏览器多账号切换测试。
3. **缓存与迁移层（localStorage / migration）**
   - 必须验证缓存 key 隔离、登出清理、迁移只作用于当前用户。

任意一层缺失，QA 视为不通过。

---

## 3. 强制测试矩阵（最小集）

### 3.1 数据库层（已有脚本）

- 运行：`qa-user-isolation` 脚本（新账号或现有账号版本均可）
- 验证：
  - A 不能读/改/删 B 的 profile/race/activities；
  - B 不能读/改/删 A 的数据。

### 3.2 前端账号切换（新增强制）

在同一浏览器、同一设备上执行：

1. 登录 A，填写并保存唯一标识数据（姓名、比赛名、备注包含 `A_ONLY_<timestamp>`）。
2. 登出 A。
3. 登录 B（此前从未使用过该浏览器更佳）。
4. 检查 Setup / Dashboard / Analysis：
   - 不得出现 A 的任何标识数据；
   - 若 B 无数据，应为空或默认态，不得继承 A 数据。
5. 切回 A 再验证 A 数据仍在。

### 3.3 缓存键与迁移逻辑（新增强制）

- 检查 localStorage：
  - 不允许使用全局 `trifit_profile|trifit_race|trifit_equipment` 作为共享键；
  - 必须使用按用户隔离键（例如 `trifit_profile:<userId>`）。
- 检查 `signOut`：
  - 至少清理当前用户缓存，不能遗留可被下一账号读取的敏感信息。
- 检查迁移：
  - `migrateLocalStorageData(userId)` 只能读取该 `userId` 对应缓存；
  - 禁止把未知来源缓存迁入当前账号。

### 3.4 失败/降级路径（新增强制）

模拟 Supabase 查询失败（断网或接口失败）后验证：

- 前端 fallback 只能读当前用户隔离缓存；
- 不得回退到跨账号共享缓存；
- 错误提示不能泄露其他用户数据。

## 4. 安全要求（强制）

### 4.1 数据隔离

- 所有用户数据查询必须显式基于当前登录用户身份约束（`user_id` 或 `id = auth.uid()` 等）。
- 前端不得以“方便”为由合并多用户缓存空间。

### 4.2 最小暴露原则

- 登出后不得在 UI 或缓存继续暴露上一个账号的个人信息。
- 历史记录、调试日志不得打印敏感个人信息明文（邮箱可脱敏显示）。

### 4.3 默认安全失败（Fail Secure）

- 当云端读取失败时，宁可显示空状态，也不能展示可能属于其他账号的数据。
- 所有 fallback 必须是“同用户、可追溯、可验证”的。

### 4.4 发布门禁

以下任一项失败，禁止合并/发布：

- 跨账号切换测试失败；
- 发现共享缓存键；
- 迁移逻辑存在跨用户读写可能；
- RLS 隔离脚本失败。

---

## 5. 每次交付时的 QA 输出模板（必须提供）

每次我完成相关改动时，必须给出：

1. **执行清单**：执行了哪些测试（数据库层 / 前端切换 / 缓存迁移 / 失败路径）。
2. **结果**：每项 PASS/FAIL。
3. **证据**：关键日志、关键页面观察点、必要的代码定位。
4. **风险结论**：是否可发布；若不可发布，阻塞项是什么。

## 6. 本文件的执行约束

- 本文件是 TriFit 项目 QA 与安全验证的强制基线。
- 后续凡涉及用户数据的任务，必须按本文件执行，不得只做单层测试。
- 若出现新的泄露类型，必须先更新本文件，再继续开发。
